# Как это работает и насколько это безопасно

---

## Архитектура: кто что делает

### Агент (на каждом агент-сервере)

Агент — это Python-скрипт в Docker-контейнере. Работает на сервере, который физически или
через VPN стоит рядом с Keenetic-роутером.

**Что делает:**
1. Каждые 60 секунд обращается к Keenetic API (`http://192.168.1.1/rci/show/...`) — это
   стандартный HTTP-эндпоинт роутера, который доступен только локально через WireGuard-туннель.
2. Собирает метрики: CPU, RAM, LTE-сигнал, трафик интерфейсов, список устройств, ARP, DHCP.
3. Делает `ping 8.8.8.8` — проверяет интернет-доступность с точки зрения этого сервера.
4. Упаковывает всё в JSON, **шифрует** и отправляет POST-запросом на центральный сервер.

**Что агент НЕ делает:**
- Не хранит данные локально (нет БД, нет файлов с метриками).
- Не открывает никаких портов — только исходящие соединения.
- Не имеет доступа к интернету кроме отправки на receiver (можно ограничить файрволом).

---

### Центральный сервер

Четыре Docker-контейнера, все в одной сети, внешу видны только два порта.

```
[Агент-серверы] ──POST:8080──► [Receiver]
                                    │
                                    ▼ пишет
                               [InfluxDB :8086]  ← только внутри Docker, снаружи не виден
                                    │
                               [Grafana :3000] ◄── твой браузер
```

**Receiver** (`receiver.py`) — Flask-сервер:
- Принимает зашифрованный payload от агентов.
- Расшифровывает, парсит JSON.
- Записывает метрики в InfluxDB через официальный клиент.
- Не хранит данные сам — только проксирует в БД.

**InfluxDB** — time-series база данных:
- Слушает только на `127.0.0.1:8086` — снаружи Docker-сети недоступна.
- Хранит все метрики с временными метками.
- Имеет собственную токен-аутентификацию.

**Grafana** — веб-интерфейс:
- Читает данные из InfluxDB через Flux-запросы.
- Отдаёт дашборды в браузер через порт `3000`.
- Защищена логином/паролем.

**Ping Monitor** (`ping_monitor.py`) — опциональный сервис:
- Пингует заданные хосты/станции, пишет RTT в InfluxDB.

## Шифрование и безопасность передачи данных

### Как работает шифрование

Данные от агента до сервера шифруются **Fernet** — это стандарт из библиотеки `cryptography`:
- Алгоритм: **AES-128-CBC** с PKCS7-паддингом.
- Аутентификация: **HMAC-SHA256** — защита от подделки и модификации.
- Ключ не передаётся — он **генерируется** из `ENCRYPTION_KEY` через **PBKDF2-HMAC-SHA256**
  с 100 000 итерациями и фиксированной солью.

```
ENCRYPTION_KEY (строка) ──PBKDF2──► 256-bit ключ ──► Fernet(AES-128-CBC + HMAC)
```

Это значит:
- Перехваченный трафик расшифровать без ключа **невозможно**.
- Подделать пакет без ключа **невозможно** (HMAC проверяется до расшифровки).
- Если `ENCRYPTION_KEY` разный на агенте и сервере — receiver вернёт 403, данные не запишутся.

### Что передаётся по сети

Только зашифрованный бинарный blob. В нём:
- `router_id` — имя роутера (задаёшь сам)
- `timestamp` — unix timestamp
- `metrics` — JSON с метриками роутера

**Пароли роутера** (`KEENETIC_PASS`) хранятся только в `.env` на агент-сервере.
На сервер они **никогда не передаются** — агент использует их только для авторизации на
роутере через локальный API.

## Насколько данные локальны

| Данные | Где хранятся | Куда уходят |
|---|---|---|
| Пароль от роутера | `.env` на агент-сервере | Никуда |
| Ключ шифрования | `.env` на агент-сервере и сервере | Никуда |
| Метрики (CPU, трафик, устройства) | InfluxDB на твоём сервере | Никуда |
| Дашборды, графики | Grafana на твоём сервере | Никуда |
| Токен InfluxDB | `.env` на сервере | Только внутри Docker-сети |

**Никакие данные не уходят в облако.** Grafana Cloud, InfluxDB Cloud, телеметрия —
всё это отключено. Используются self-hosted версии. Единственный внешний запрос при работе —
`ping 8.8.8.8` (проверка интернета, выполняется с агент-сервера).

## Что открыто в интернете

| Порт | Сервис | Риск | Рекомендация |
|---|---|---|---|
| `8080` | Receiver | Низкий — принимает только зашифрованные данные, без ключа бесполезен | Ограничить IP-адресами агент-серверов |
| `3000` | Grafana | Средний — веб-интерфейс с паролем | Поставить надёжный пароль или закрыть за VPN |
| `8086` | InfluxDB | **Не открыт** — только внутри Docker | Не открывать никогда |

Если receiver доступен всем — злоумышленник может слать мусорные зашифрованные запросы,
receiver будет пытаться их расшифровать и возвращать 403. Это не опасно, но создаёт нагрузку.
Поэтому рекомендуется ограничить `8080` через файрвол:
```bash
ufw allow from AGENT-IP to any port 8080
ufw deny 8080
```

## Что получаешь в результате

- **Единая точка мониторинга** для любого количества Keenetic-роутеров с LTE.
- **История метрик** — видно как менялся сигнал, трафик, нагрузка за любой период.
- **Оповещение об офлайн** — если роутер пропал, статус сразу меняется на "ОФЛАЙН".
- **Список устройств** — кто сейчас в сети, сколько трафика потребил каждый.
- **Интернет-доступность** — отличие "роутер упал" от "интернет упал" (SIM-карта, оператор).
- **LTE-качество** — RSSI, RSRP, RSRQ, SINR в динамике, видно деградацию сигнала.
- **WireGuard** — трафик туннеля по пирам.

## Потребление трафика туннеля

Один опрос (9 эндпоинтов Keenetic API + ping) ≈ **15–30 KB**.

| Интервал | Трафик/час | Трафик/месяц |
|---|---|---|
| 60 сек (по умолчанию) | ~1.5 MB | ~1.1 GB |
| 120 сек | ~750 KB | ~540 MB |
| 300 сек | ~300 KB | ~216 MB |

На LTE не ощутимо. На каждый дополнительный роутер — умножить.

## Зависимости и доверие

| Компонент | Откуда | Доверие |
|---|---|---|
| Python 3.13-slim | AWS ECR Public (зеркало Docker Hub) | Официальный образ |
| InfluxDB 2.7 | Docker Hub | Официальный образ InfluxData |
| Grafana 10.2 | Docker Hub | Официальный образ Grafana Labs |
| `cryptography` | PyPI | Стандартная библиотека шифрования для Python |
| `flask` | PyPI | Стандартный веб-фреймворк |
| `influxdb-client` | PyPI | Официальный клиент InfluxData |
| `requests` | PyPI | Стандартная HTTP-библиотека |

Сторонних облачных сервисов нет. Код агента и receiver — ~500 строк Python, можно прочитать
за 10 минут.
